# Pixpeel 憲法

## 核心原則

### I. 品質優先

每一行程式碼都必須是可維護、可讀、且正確的。品質不是事後補救，而是開發過程中的內建要求。

- 程式碼 MUST 清晰表達意圖，避免不必要的縮寫或魔法數字
- 變數、函式、模組命名 MUST 自我描述，不需額外注解才能理解
- 重複程式碼（DRY 違反）MUST 在同一 PR 中提取或說明保留原因
- 每次 PR 合併前 MUST 通過 linting 與格式化檢查

**理由**：低品質的程式碼在短期省時，卻在長期製造債務。MVP 不等於劣質品。

### II. 測試驅動開發（不可妥協）

功能實作 MUST 先有測試，再有程式碼。

- 開發順序 MUST 為：撰寫測試 → 確認測試失敗（紅燈）→ 實作功能 → 確認測試通過（綠燈）→ 重構
- 每個用戶故事 MUST 有對應的驗收測試（Acceptance Test）
- 單元測試覆蓋率 MUST 涵蓋所有業務邏輯路徑
- 整合測試 MUST 覆蓋跨模組的關鍵流程
- 測試 MUST 獨立執行，不依賴執行順序或外部狀態

**理由**：可被測試的設計即是良好設計。測試是活文件，也是重構的安全網。

### III. MVP 優先

每個功能 MUST 先交付最小可行版本，驗證後才擴展。

- 功能設計 MUST 識別 P1（最核心）用戶故事，並能在不實作 P2/P3 的情況下獨立交付
- MVP 交付後 MUST 經過驗證（演示或測試通過）才能開始下一優先級
- 新功能需求 MUST 先問：「這是 MVP 所必需的嗎？」，否則列入 backlog
- P1 故事 MUST 在 P2 故事開始前完整測試通過

**理由**：更快取得真實回饋、降低浪費、確保每次增量都能獨立交付價值。

### IV. 簡約設計

設計 MUST 只解決當前已知的問題，不為假設的未來需求預留架構。

- 禁止為「以後可能需要」的功能建立抽象層或介面
- 三個相似的程式碼段 MUST 考慮提取；兩個相似的不必須提取
- 每新增一個依賴、框架、或設計模式 MUST 有明確的理由
- YAGNI（You Aren't Gonna Need It）原則 MUST 作為設計決策的預設立場
- 複雜度 MUST 被追蹤並在 plan.md 的「複雜度追蹤」表格中說明

**理由**：過度設計是 MVP 的最大敵人。簡單的系統更容易測試、更容易維護。

### V. 獨立可測試

每個用戶故事（User Story）MUST 能獨立實作、獨立測試、獨立演示。

- 用戶故事 MUST 定義明確的「獨立測試」說明（Independent Test）
- 功能邊界 MUST 清晰，避免用戶故事之間隱性耦合
- 每個故事完成後 MUST 設置檢查點（Checkpoint）並驗證獨立功能
- 模組 MUST 具有明確的輸入輸出介面，不依賴全域狀態

**理由**：獨立可測試是 MVP 遞增交付的技術前提，也是快速除錯的基礎。

## 開發工作流程

- 功能開發 MUST 從 `spec.md` 開始，定義用戶故事與驗收條件
- 實作前 MUST 完成 `plan.md`，確認技術上下文與架構決策
- 任務拆分 MUST 依用戶故事分組，確保每個故事可獨立完成
- 程式碼合併 MUST 附帶對應測試通過的證明
- 重大架構改動 MUST 更新 `plan.md` 並記錄決策理由

## 品質閘門

以下條件 MUST 全部滿足，功能才可視為完成：

- [ ] 所有測試通過（包含新增的測試）
- [ ] 沒有 linting 錯誤
- [ ] 用戶故事的驗收測試通過
- [ ] MVP 用戶故事（P1）可獨立演示
- [ ] 無未說明的複雜度（若有，記錄於 plan.md）
- [ ] 程式碼可讀性符合品質優先原則

## 治理規則

本憲法優先於所有其他開發慣例與個人偏好。

- 憲法修訂 MUST 以 PR 形式提交，並更新版本號與修訂日期
- 版本號遵循語義化版本規則：MAJOR（原則移除/重定義）、MINOR（新增原則/章節）、PATCH（措辭修正）
- 所有 PR 審查 MUST 確認本次變更符合憲法原則
- 若有充分理由需要違反原則，MUST 記錄於 plan.md 的「複雜度追蹤」表格並取得共識
- 憲法本身每季 SHOULD 回顧一次，確認原則仍符合專案現況
